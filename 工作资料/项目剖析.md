# 项目剖析

## Oneapplication

### 基本情况

项目人数： 研发 4 人， 测试  2 人

两位同事负责： 路由 Camel 开发

我负责： 用户历史还款记录 + 工商信息模块 + 整个模块的整合

另一名同时负责：规则优化和修改

项目进度：今年3月份上线，目前是迭代阶段



### 项目的主要特色

####  路由平台

**功能方面：**贷款申请信息的整合

**路由平台的角色：**路由平台对于多个模块运行流程的统一，个人理解是更加类似于设计模式中的模版方法，但是这个Camel路由的功能会更加全面一些，

比如：每个执行流程的监控、配置化流程的控制、多线程调用、流程同步与异步调用方式的处理

举个例子：

整个流程分为四个步骤：

1. 解析原始数据信息，提取申请访问信息
2. 接口调用
3. 保存接口返回信息
4. 数据处理方法

这四个步骤，可用通过配置实现流程的转变，或者同步、异步的方式去执行



#### Token 认证

场景： 这个项目会涉及到工商信息对的调用，所以加入了 token 认证，用户会得到一个 ID 以及 密钥，通过ID 和密钥申请 时效2小时的Token，在访问的过程中，需要携带Token，系统会对Token进行验证，是否有效、且在有效的时间内





### 项目中的问题点

#### 1. 并发死锁问题

**现象**：同样的报文，反复调用这个系统的过程，会有并发死锁的问题

**原因**：数据库主键是报文的申请编号，当同一笔报文反复调用系统的时候，由于是多线程方式运行，就可能存在多个线程同时处理同一个报文的情况，由于数据库存储的时候，会涉及到删除原始数据，所以会存在多个线程操作数据库资源的问题，从而造成了死锁，后来的解决方案是，在数据库读写这一步加入 synchronized 关键字





### 计算逻辑：

1. 历史还款记录的不良记录
2. 工商信息中：失信人、公司诉讼、财产冻结等信息提取





## PBOC Summary 

### 基本情况

项目人数： 研发 3 人， 测试 5人

一位同事负责： 主系统提供一个征信报告的信息

另一名同事负责规则的统一：比如不良资产、负债、行用卡欠款、历史欠款次数、短时期内申请贷款的频率这些

我负责：汇准报告 + 信息删除模块 + 整个模块的整合



### 项目的主要特色

**消息队列解耦、流量削峰：** 这个项目独立运行，由于涉及到网络磁盘的读写，所以处理性能上会比较慢，当流量比较大的时候，消息队列也可以起到一个流量削峰的作用。

**并发容器的使用：** 删除数据模块会有多线程分别去删除数据库、磁盘数据文件、磁盘的日志文件，并发容器的功能就是对于删除过程中如果某一条数据删除失败会做一个记录的功能，不影响其他数据继续进行删除，同时最终对于删除结果进行一个统计

# 各类中间件

## 消息队列

### 功能使用

1. PBOC Summary 中 解耦、 流量削峰
2. 订单支付功能中使用到了：延时队列
3. 分布式事务：可靠消息+最终一致性



### 缓存 

1. 短信有效期
2. 订单模块，使用Redis实现数据的存储，因为涉及到高频的读写
3. 页面缓存：首页部分、因为首页展示的内容会比较多，所以针对高频访问的地方，会直接用Redis来实现高性能写服务
4. 还有类似的分布式锁、
5. 布隆过滤器：用来快速判断一个字符串是否存在、布隆过滤器可以用于检索一个元素是否在一个集合中
6. 结合Spring Session 实现共享 Session 
7. 限流 ： GateWay + Redis 实现限流





### 数据库Sql优化方面

业务解决方案，是需要建立在场景上面的的：

1. 一次性需要返回大量需要的数据，比如首页页面，那这个时候，就需要使用到缓存
2. Sql语句写的有问题
   1. 一次性返回大量不需要的数据
   2. Sql 写的有问题，没有使用到索引
   3. 数据库设计有问题，没有添加合适的索引
   4. 数据库数据量太大，这个时候需要分库分表 或者 使用搜索殷勤







### Sentinel

流量控制、熔断降级、系统负载保护
